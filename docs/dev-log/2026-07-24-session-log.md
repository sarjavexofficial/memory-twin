# 開発ログ 2026-07-24（★根本原因解決: 有料App契約の保留＝W-8BEN未完成）

## 結論: 6日間の code=23 の真犯人は「有料App契約が未有効（ユーザ情報を保留中）」だった

7/18から続いていた「プラン画面で価格が出ない／RevenueCat ERR code=23（StoreKitが商品0件）」の
真の原因が判明し、**完全解決**した。反映待ちでも設定ミスでもなく、**有料App契約が有効になっていなかった**。

### 判明した事実（App Store Connect「ビジネス > 契約」で確認）
- **有料アプリ契約のステータス = 「ユーザ情報を保留中」**（7/18に「有効」と誤認していたが、実際は保留のまま6日間）
- **U.S. Form W-8BEN = 「税金情報が不足しています」** ← 未完成だった
  - 8. Date of Birth（生年月日）が空欄
  - 9. 日米租税条約の居住者チェックが未チェック
- 銀行口座 sarjavex (9061) = 「処理中」

有料App契約が保留状態だと、**IAP商品はStoreKitに一切流れない**。だから何日待っても、
何度再読み込みしても、Appleに問い合わせても、アプリを直しても、絶対に価格は出なかった。

### 解決手順（ゆずが操作・Claudeがガイドと確認）
1. ゆずがApp Store Connectにログイン → Claudeが契約ページを確認して原因特定
2. W-8BENを完成: 生年月日 04-05-2003、9番の租税条約チェック、Part III署名認証
3. ゆずが「送信」（税務書類のため署名・送信は本人が実施）
4. → **全項目が「有効」に変化**（契約・銀行口座・W-8BEN すべて有効。送信日7/24）

### 今度こそ「正常な反映待ち」
契約がActiveになったので、Appleが商品をStoreKitに流し始める。反映は数時間〜最大24h。
これまでと違い、今度は確実に価格が出る。出たら購入テスト→スクショ→審査提出。

### 学び（最重要・再発防止）
- **「有料App契約が有効」を必ずステータス文言で確認すること。** 契約に同意しただけでは
  「ユーザ情報を保留中」であり、税務フォーム＋銀行口座の両方が「有効」になって初めて契約がActiveになる
- W-8BENは「追加」しただけでは不完全。生年月日・租税条約チェック・署名まで完了して「有効」になる
- code=23（ConfigurationError）は設定が正しくても契約未有効で出る。今回のように
  ASC API側の商品・RevenueCat側が全てシロでも、契約保留なら商品0件になる
- 診断の遠回り: RevenueCat/StoreKit/端末/反映待ちを延々調べたが、真因は「契約ページの保留警告」。
  IAPが出ない時は**最初に契約ステータスを文言で確認**すべきだった
